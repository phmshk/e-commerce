# Предложение по масштабируемой архитектуре e-commerce проекта

На текущий момент проект использует классическую структуру Next.js (App Router), где большинство компонентов, моделей и типов сосредоточены внутри директории `app/`. Для небольшого проекта это допустимо, но с ростом функционала (корзина, заказы, личный кабинет, админка) такая структура может привести к сильной связанности кода и трудностям в поддержке.

## Текущие болевые точки
1. **Смешивание уровней**: Компоненты UI, бизнес-логика и модели данных находятся в одной папке `app/`.
2. **Плоская структура компонентов**: Папка `app/components` содержит и базовые UI-элементы, и сложные блоки страниц.
3. **Отсутствие явных границ модулей**: Трудно отследить зависимости между товарами, постами и пользователями.

---

## Предлагаемая архитектура: Feature-Sliced Design (FSD) или Модульный подход

Я предлагаю вынести основную логику из `app/` в корневую директорию `src/` (опционально) или организовать структуру внутри `app/` по принципу разделения ответственности.

### Рекомендуемая структура папок:

```text
/
├── app/ (Только роутинг и конфигурация страниц)
│   ├── api/
│   ├── (auth)/
│   ├── (shop)/
│   │   ├── products/
│   │   └── sale/
│   ├── layout.tsx
│   └── page.tsx
├── src/ (или в корне)
│   ├── shared/ (Общие переиспользуемые компоненты и утилиты)
│   │   ├── ui/ (Button, Input, Badge - текущий shadcn)
│   │   ├── api/ (Базовые инстансы для запросов)
│   │   ├── lib/ (utils.ts, dbConnect.ts)
│   │   └── types/ (Общие типы)
│   ├── entities/ (Бизнес-сущности: Product, Post, User, Order)
│   │   ├── product/
│   │   │   ├── ui/ (ProductCard, ProductList)
│   │   │   ├── model/ (Mongoose Schema, types)
│   │   │   └── api/ (Методы получения данных о продукте)
│   │   └── post/
│   │       ├── ui/ (PostCard, PostGrid)
│   │       ├── model/ (Post Schema, types)
│   │       └── api/
│   ├── features/ (Действия пользователя: AddToCart, Search, Login)
│   │   ├── product-search/
│   │   └── auth-by-email/
│   └── widgets/ (Крупные блоки: Header, Footer, HeroCarousel)
│       ├── header/
│       ├── footer/
│       └── product-grid/
```

## Основные изменения и преимущества:

### 1. Вынос UI-библиотеки (Shadcn)
Переместить `app/shadcn/components/ui` в `shared/ui`. Это подчеркнет, что данные компоненты являются "глупыми" и не зависят от бизнес-логики.

### 2. Организация сущностей (Entities)
Вместо общих папок `app/models` и `app/types`, разбить их по бизнес-смыслу.
- Например, всё, что касается продукта (схема Mongoose, типы TS, базовые карточки), живет в `entities/product`. Это упрощает поиск кода и его тестирование.

### 3. Слой Виджетов (Widgets)
Текущие компоненты из `app/components/main/blocks` (BlockSales, BlockNewArrivals) логично перенести в `widgets`. Это композиционные слои, которые объединяют сущности и фичи.

### 4. Изоляция бизнес-логики (Services/Actions)
Для работы с базой данных (Mongoose) рекомендуется создать слой сервисов или использовать Server Actions, отделенные от UI-компонентов. Это позволит избежать прямого импорта моделей в компоненты (что сейчас частично наблюдается).

### 5. Улучшение типизации
Разделить типы на:
- **Domain Types**: Описание данных из БД (модели).
- **DTO (Data Transfer Objects)**: Описание данных, передаваемых через API.
- **UI Types**: Пропсы компонентов.

## Рекомендации по реализации:
1. **Постепенный рефакторинг**: Не обязательно переносить всё сразу. Можно начать с выноса `shared/ui` и `entities`.
2. **Абсолютные импорты**: Настроить `tsconfig.json` для использования алиасов (напр. `@/entities/*`, `@/shared/*`).
3. **Server Components**: Максимально использовать возможности Next.js, оставляя логику получения данных на сервере в `entities/*/api`.

Это предложение позволит проекту масштабироваться без превращения папки `components` в "свалку", а также упростит командную разработку.
